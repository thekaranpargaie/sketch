# Requirements: Sketch UI/UX Fixes & DTO Bug Patch

**Feature Folder**: `docs/features/sketch-ui-ux-fixes/`  
**Strategist**: GitHub Copilot  
**Status**: Draft — Pending Orchestrator Approval  
**Date**: 2026-03-31  
**Priority**: High (Bug: P0 / UI: P1 / Tutorial: P1)

---

## 1. Problem Statement

Three independent issues degrade the quality of the Sketch application (a visual blueprint-to-code scaffolding tool):

| # | Issue | Impact |
|---|-------|--------|
| 1 | Scaffolding engine generates syntactically invalid C# DTO records | Generated code does not compile — core product failure |
| 2 | UI is visually minimal and lacks professional polish | Reduces product credibility and user confidence |
| 3 | No onboarding guidance, connection hints, or tutorial | New users cannot discover the connection rules or understand the tool |

---

## 2. Scope & Out-of-Scope

### In Scope
- Fix the C# DTO generation bug in `ScribanRenderer.RenderDto()`
- Redesign node cards, the toolbar palette, and the top header bar
- Add per-node connection hints (tooltip/badge)
- Add a fixed tutorial button (`?`) that opens an onboarding modal
- The modal includes step-by-step instructions, connection rule reference, and a sample app diagram

### Out of Scope
- Changing the backend scaffolding output for Entity, Service, or Controller templates
- Altering database schemas, API contracts, or the `GenerationPlan` model
- Adding new node types or new connection semantics
- Authentication, CI/CD, deployment changes

---

## 3. User Stories

### Issue 1 — DTO Syntax Bug

**US-1.1** — Valid C# DTO Output  
*As a developer using Sketch,*  
*I want the generated DTO records to be syntactically valid C#,*  
*So that I can open the downloaded ZIP and build it without manually editing generated files.*

### Issue 2 — UI Polish

**US-2.1** — Professional Node Cards  
*As a user building a blueprint,*  
*I want node cards to have a polished appearance with clear iconography, type labels, and structured field lists,*  
*So that the canvas feels like a professional design tool rather than a prototype.*

**US-2.2** — Descriptive Toolbar Palette  
*As a new user,*  
*I want each palette item in the sidebar to show the node type name and a brief one-line description,*  
*So that I can understand what each node type does before dragging it onto the canvas.*

**US-2.3** — Improved Color Scheme and Spacing  
*As a user,*  
*I want consistent spacing, refined color tokens, and clear visual hierarchy throughout the app,*  
*So that the tool is pleasant and easy to navigate.*

### Issue 3 — Tutorial, Hints & Sample Preview

**US-3.1** — Per-Node Connection Hints  
*As a user placing nodes on the canvas,*  
*I want to see a small hint on each node (e.g., a badge or tooltip) that describes what node types it can connect to,*  
*So that I know which connections are valid without reading external documentation.*

**US-3.2** — Tutorial Button  
*As a first-time user,*  
*I want a persistent `?` button fixed at the bottom-right corner of the canvas,*  
*So that I can access onboarding help at any time without leaving the current canvas.*

**US-3.3** — Tutorial Modal: Step-by-Step Instructions  
*As a first-time user opening the tutorial,*  
*I want a modal with numbered step-by-step instructions for how to use Sketch,*  
*So that I can build my first blueprint successfully without prior knowledge.*

**US-3.4** — Tutorial Modal: Connection Rules Reference  
*As a user,*  
*I want the tutorial modal to include a clear table or diagram explaining the connection rules (what connects to what),*  
*So that I understand the semantics of each edge type.*

**US-3.5** — Tutorial Modal: Sample App Preview  
*As a user,*  
*I want the tutorial to include a worked example showing Employee, Department, and Role entities connected to a REST Protocol and a SqlServer Storage node,*  
*So that I can see a complete blueprint and understand the expected generated output.*

## 4. Acceptance Criteria

### AC-1: DTO Syntax Bug Fix

- **AC-1.1** — When `ScribanRenderer.RenderDto()` processes field lines generated by `ScaffoldingEngine` (which contain the suffix `{ get; set; }`), the output DTO record parameters must contain **no property accessors**. The replacement must strip `" { get; set; }"` (not `" { get; init; }"`).
- **AC-1.2** — The generated `UserDto` for a User entity with fields `(Id: Guid, Name: string, Email: string)` must exactly match positional record syntax:
  ```csharp
  public record UserDto(
      Guid Id,
      string Name,
      string Email);
  ```
- **AC-1.3** — The same fix applies to `CreateUserRequest` and `UpdateUserRequest` records within the same rendered file.
- **AC-1.4** — `RenderEntity()` is **not affected** — entity classes retain `{ get; set; }` property accessors (this is correct behaviour and must not be changed).
- **AC-1.5** — All existing unit tests in `Sketch.UnitTests` continue to pass after the fix.
- **AC-1.6** — The fix is a **targeted single-line change** per positional record template expression in `RenderDto()` — no other rendering methods are altered.

### AC-2: UI Polish

- **AC-2.1** — Node cards use a two-row layout: a header row with SVG/icon + bold entity name, and a body section listing fields.
- **AC-2.2** — Each node type has a visually distinct color accent that aligns with the dark theme (deep slate `#0f172a` background):
  - Entity: blue accent (`#3b82f6` / `#1d4ed8`)
  - Protocol: violet accent (`#8b5cf6` / `#6d28d9`)
  - Storage: amber accent (`#f59e0b` / `#b45309`)
  - Identity: emerald accent (`#10b981` / `#047857`)
- **AC-2.3** — The `NodeToolbar` sidebar is at least 180px wide and each palette card shows: icon, type name (bold), and a one-line description (muted text below the name).
- **AC-2.4** — The top header bar includes the Sketch logo/wordmark, a project name editor, and action buttons (Clear, Provision) with consistent button styling (rounded, hover states visible).
- **AC-2.5** — Font rendering uses `Inter`, `system-ui`, or equivalent. Minimum body font size is 13px; headings within nodes are at minimum 14px bold.
- **AC-2.6** — Spacing between nodes, handles, and text follows an 8px base grid.
- **AC-2.7** — Selected node state is visually distinct: highlighted border, subtle glow or shadow — not just a color shade change.

### AC-3: Hints, Tutorial Button, and Modal

- **AC-3.1** — Each node card displays a small connection hint, visible on hover or as a persistent muted badge in the node footer. The hint text follows these rules:
  - Entity node: `"Connects to → Protocol, Storage, Identity"`
  - Protocol node: `"Source only — drag to Entity nodes"`
  - Storage node: `"Source only — drag to Entity nodes"`
  - Identity node: `"Connects to → Entity nodes"`
- **AC-3.2** — A round `?` button (diameter ≥ 44px, accessible touch target) is fixed at `bottom: 24px; right: 24px` on the canvas page, always visible above all canvas content (`z-index` above ReactFlow).
- **AC-3.3** — Clicking the `?` button opens a modal overlay. The modal is closeable via an `×` button and by clicking outside the modal (backdrop click).
- **AC-3.4** — The modal contains three clearly labelled sections:
  1. **Getting Started** — numbered steps (1–5) covering: add nodes, name them, connect them, fill properties, click Provision.
  2. **Connection Rules** — a table with columns: Node Type | Can Connect To | Description.
  3. **Sample Blueprint** — a static diagram or structured preview showing the Employee/Department/Role example (see §5).
- **AC-3.5** — The modal is scrollable if content exceeds viewport height.
- **AC-3.6** — The modal is keyboard accessible: `Escape` closes it; focus is trapped inside while open.
- **AC-3.7** — The sample blueprint preview clearly labels each entity's fields and shows the REST Protocol and SqlServer Storage connections in a visual or table format.

## 6. Edge Cases & Constraints

### Bug Fix Edge Cases
- **EC-1.1** — An entity with zero fields must still produce an empty but syntactically valid positional record: `public record EmptyDto();`
- **EC-1.2** — Fields whose types contain angle brackets (e.g., `List<string>`) must not have their content corrupted by the string replacement.
- **EC-1.3** — The strip logic operates only on DTO template rendering; entity class rendering (`RenderEntity`) is unaffected.

### UI Constraints
- **EC-2.1** — The canvas (ReactFlow area) must not lose any existing drag-and-drop, connection, or selection behaviour after visual restyling.
- **EC-2.2** — The PropertiesPanel behaviour (shown only when a node is selected) must remain unchanged.
- **EC-2.3** — All existing inline `style` objects may be refactored to a consistent `tokens` or style-constant pattern, but CSS-in-JS or Tailwind are both acceptable — no new dependencies should be added unless already present in `package.json`.

### Tutorial Constraints
- **EC-3.1** — The tutorial modal is purely presentational (read-only). It must not interact with or modify the Zustand blueprint store.
- **EC-3.2** — The `?` button must not overlap or obscure the Provision button or Toast notifications when both are visible.
- **EC-3.3** — The tutorial component must be a standalone React component created in `src/Sketch.Web/src/components/Tutorial/` — not inlined inside `CanvasPage`.
